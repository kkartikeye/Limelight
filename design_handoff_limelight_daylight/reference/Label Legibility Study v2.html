Limelight · design note · label legibility · v2 (post-code-review)

Country labels on the news-intensity heatmap

After reading the repo: you already apply #f0e0c8 warm parchment + rgba(0,0,0,0.92) 2.5px halo to the country-label layer. That’s the right floor, but it's still one color across a 9-stop YlOrRd palette that ranges from cream to near-black, plus a no-data dark and the space background. A single ink will always trade legibility on one end for the other. The fix is a data-driven label color derived from the same score that drives the fill.

v2 changes · effective palette computed from fill-opacity 0.8 over dark-v11 basemap · Mapbox-native code, not CSS · score-bucketed recommendation

Read directly from kkartikeye/Limelight@main. Your LABEL_STYLES table is well-considered — the per-layer color hierarchy (countries warmer than cities warmer than water) is a nice touch. The issue is purely with country-label being a single color across all heat fills.

Where it still struggles

Single-color failure modes

  • Cream/yellow countries (score 0–25): the parchment #f0e0c8 sits inside the YlOrRd yellow family. Halo carries the contrast alone — legible but visually muddy.
  • Dark red countries (score 75+): parchment-on-dark-red works, but the label loses semantic weight against the alarming fill.
  • No-data countries on the dark basemap: parchment works, halo is wasted.
  • Region labels (EUROPE / ASIA / AFRICA) inherit the same color — they should be visibly demoted.

02The effective on-screen palette

Your heat layer paints YlOrRd at fill-opacity 0.8 over the dark-v11 basemap (≈#0c0c12). The labels don't sit on the raw ColorBrewer stops — they sit on these mixed tones, which run noticeably darker than the source. Recommendations need to test against these, not the raw palette.

Drive label color from the score you already have

Why bucketed, not continuous

A continuous luminance-driven color is tempting but reads as visual noise — every country’s label ends up a slightly different gray. Three buckets aligned to the YlOrRd palette’s natural break (yellows → reds at the score-50 hinge) gives you a legible system, not just legible labels.

The labels need to come off your countries source

Mapbox’s basemap country-label layer doesn’t know about your scores, so a data-driven text-color expression there has nothing to read. Build a sibling symbol layer on the countries source you already maintain — one line in heat-layer.tsx — then hide the basemap’s country labels.

Drop into heat-layer.tsx (after the heat-fill / heat-outline blocks)

// Score-bucketed text color & halo — reads same score as the fill.
// 0–50  → near-black ink + cream halo (light heat tones)
// 62+   → warm-white ink + dark halo (deep heat tones)
// no-data → warm parchment + dark halo (basemap shows through)
const LABEL_TEXT_COLOR: mapboxgl.Expression = [
  "case",
  ["==", ["coalesce", ["get", "score"], 0], 0], "#f0e0c8", // no data
  ["<",  ["get", "score"], 55],                "#1a140a", // light bucket
  "#fff3df"                                                  // dark bucket
];

const LABEL_HALO_COLOR: mapboxgl.Expression = [
  "case",
  ["==", ["coalesce", ["get", "score"], 0], 0], "rgba(0,0,0,0.92)",
  ["<",  ["get", "score"], 55],                "rgba(240,224,200,0.85)",
  "rgba(0,0,0,0.92)"
];

map.addLayer({
  id: "country-label-custom",
  type: "symbol",
  source: "countries",
  layout: {
    "text-field": ["get", "ADMIN"],
    "text-font": ["DIN Pro Medium", "Arial Unicode MS Regular"],
    "text-size": ["interpolate", ["linear"], ["zoom"],
                  2, 11,  4, 13,  6, 15],
    "text-letter-spacing": 0.02,
    "text-max-width": 7,
    "symbol-sort-key": ["-", 100, ["coalesce", ["get", "score"], 0]], // prioritise hot
  },
  paint: {
    "text-color": LABEL_TEXT_COLOR,
    "text-halo-color": LABEL_HALO_COLOR,
    "text-halo-width": 1.6,
    "text-halo-blur": 0.3,
  },
});

// Suppress the basemap country labels so they don't double up
if (map.getLayer("country-label")) {
  map.setLayoutProperty("country-label", "visibility", "none");
}
Why 55, not 50

The bucket break

Sets the light/dark crossover slightly above the YlOrRd mid-point so the #fd8d3c orange (score 50) still gets dark ink — that tone is borderline either way and dark ink reads as more confident on it. Tune to taste; 55 is a defensible default.

Why hide basemap labels

Avoid double-stamping

If you leave the basemap’s country-label visible alongside the custom layer, Mapbox will render both and collide them. Suppress the basemap layer once your custom layer is in place; for countries with no heat-fill, the custom layer still draws because the source includes every feature.

Region labels

Demote continents

EUROPE / ASIA / AFRICA in your screenshot are the basemap’s continent-label layer — not in your LABEL_STYLES dict. Add it and drop opacity to ~0.5; they should read as quiet typographic chrome, not as data.

Ocean labels

Italic, no halo, dim

Add a water-label entry to LABEL_STYLES: #9ab4c8, halo width 1. Oceans sit on the dark space background where halo just adds fuzz around perfectly-contrasted text.

If you only have 5 minutes — ship C instead

// In LABEL_STYLES, country-label entry:
"country-label": {
  color: "#f4ecd8",              // slightly cooler warm-white
  haloColor: "rgba(0,0,0,1)",   // from 0.92 → 1.0
  haloWidth: 3.5,                 // from 2.5 → 3.5
},

That single change closes the gap on the no-data and hot-red countries. It won’t make labels feel like they belong to their fills the way B does — but it stops them from being invisible.
